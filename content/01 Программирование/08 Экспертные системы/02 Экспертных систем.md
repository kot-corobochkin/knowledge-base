
## 4. Expert Systems

Expert systems are a core application of AI designed to simulate the judgment and behavior of a human expert.
- **Definition:** A computer program that simulates the judgment and behavior of a human expert in a particular field.
- **Components:**

**Knowledge Base:** A repository of accumulated experience.
- **Set of Rules:** Applied to each specific situation to reach a conclusion.

- **Key Roles:**

**Knowledge Engineer:** An individual who studies how human experts make decisions and translates that expertise into rules a computer can understand.

- **Other Names:** Knowledge-based systems, rule-based systems.
- **EMYCIN:** One of the first expert system "shells," derived from the MYCIN medical diagnosis system. A shell allows for the creation of new expert systems without building the inference engine from scratch.
- **Purpose:** To perform tasks that normally require human expertise, such as narrowing down a set of symptoms to a likely subset of causes.

Source: **Book:** _Artificial Intelligence and Expert Systems_
**Chapter:** Chapter 2, "Applications of Artificial Intelligence"

## 1. Introduction to Expert Systems

Expert systems are among the first truly successful forms of AI software, first created in the 1970s and proliferating in the 1980s. They represent a novel approach called **rule-based systems**.

**Key Definitions:**
- A computer system that simulates the decision-making ability of a human expert.
- A program that contains a knowledge base of accumulated experience and a set of rules for applying that knowledge to specific situations.
- An intelligent program that uses domain-specific knowledge to solve complex problems.

**Key Terminology:**
- **Knowledge Engineer:** An individual who studies how human experts make decisions and translates that expertise into rules a computer can understand.
- **Knowledge Engineering:** The process of developing an expert system.
- **EMYCIN:** One of the first expert system "shells," derived from the MYCIN medical diagnosis system.

## 2. Features of Good Expert Systems

A well-designed expert system should be:

| Feature | Description |
| --- | --- |
| Useful | Developed to meet a specific, documented need |
| Usable | Easy to use, even for novice computer users |
| Educational | Allows non-experts to increase their expertise |
| Explanatory | Able to explain its reasoning process |
| Capable of Learning | Able to acquire new knowledge through questioning |
| High-Performing | Provides high-quality output |
| Timely | Produces decisions within acceptable time constraints |
| Heuristic | Uses heuristics to narrow the search space |
## 3. Architecture and Components of Expert Systems

The core architecture of a rule-based expert system consists of several interconnected components.

## 3.1 User Interface
- Provides communication between the user and the system.
- Can be text-oriented or graphical (graphical is more user-friendly).
- Allows users to enter commands, respond to questions, and see explanations of conclusions.

## 3.2 Knowledge Base

The **heart** of the expert system. Contains domain-specific knowledge acquired from human experts.

**Types of Knowledge:**

| Type | Description |
| --- | --- |
| Procedural Knowledge | Step-by-step information for solving problems |
| Factual Knowledge | Widely shared knowledge from textbooks and journals |
| Heuristic Knowledge | Experiential, individualistic knowledge from good practice and judgment |

**Rule Structure:**
- **IF part (Antecedent/Premise):** Condition that must be satisfied
- **THEN part (Consequent/Action):** Conclusion or action taken when condition is met

**Examples of Rule Types:**

| Rule Type | Example |
| --- | --- |
| Relation | IF fuel tank is empty THEN car is dead |
| Recommendation | IF season is autumn AND sky is cloudy AND forecast is drizzle THEN take an umbrella |
| Directive | IF car is dead AND fuel tank is empty THEN refuel the car |
| Heuristic | IF spill is liquid AND pH 
**Compound Rules:** Can use AND or OR to combine multiple antecedents.

**Knowledge Base Development Processes:**
- Knowledge acquisition
- Knowledge analysis and representation
- Knowledge validation
- Inference design
- Explanation and justification

## 3.3 Working Storage (Database)
- Contains facts specific to the current problem being solved.
- Stores user answers, external data, intermediate results, and conclusions reached.

**Difference from Knowledge Base:**
- **Knowledge Base:** Permanent knowledge used repeatedly
- **Database:** Temporary data about a particular case

## 3.4 Inference Engine

The **reasoning component** (rule interpreter) of the expert system.

**Functions:**
- Matches facts against production rules (pattern matching)
- Determines which rules to fire (conflict resolution)
- Generates inference chains (how rules are applied to reach conclusions)
- Prioritizes rules and fires those with highest priority

**Conflict Resolution Strategies:**
- **Priority-based:** Rules stored in order of priority; highest priority fires first
- **Specificity-based:** Most specific rule (with more conditions) fires first

## 3.4.1 Forward Chaining (Data-Driven)

**Direction:** Facts → Conclusion

**Process:**
- Starts with known facts
- Rules are evaluated; when conditions match, rules fire
- Each firing adds new facts to working memory
- Continues until no more rules can fire

**Algorithm:**

```
while (no new assertion made) and (unresolved)
    for each rule (and for each possible binding)
        try to support rule's conditions from known facts
        if all supported then assert consequent
```

**Advantage:** Good when all facts are known
**Disadvantage:** May fire rules that don't contribute to the goal (inefficient for single-goal problems)

## 3.4.2 Backward Chaining (Goal-Driven)

**Direction:** Goal → Facts

**Process:**
- Starts with a goal (hypothesis)
- Searches for rules that conclude that goal
- If rule found, attempts to prove its conditions (sub-goals)
- Recursively continues until all conditions are satisfied or no more rules exist

**Algorithm:**

```
while (no untried hypothesis) and (unresolved)
    for each hypothesis
        for each rule with the hypothesis as the consequent
            try to support the rule's conditions from known facts or via recursion
            if all are supported, then assert the consequent
```

**Example:** Prolog is a backward-chaining language

**Advantage:** Efficient for goal-oriented problems
**Disadvantage:** May explore irrelevant sub-goals

## 3.5 Explanation Facility
- Provides reasoning behind the system's conclusions
- Displays rules used during inference
- Essential for user trust and for knowledge engineers debugging the system
- Limitations: Often explains *what* rules fired, not *why* the knowledge is correct

## 3.6 Knowledge Acquisition Facility
- Provides an automatic way for experts to enter knowledge
- Reduces reliance on knowledge engineers for coding rules

## 3.7 External Interface
- Allows the expert system to exchange data with external sources
- Handles input from data files and output to various destinations

## 5. Expert System Shells

A **shell** is a pre-built software package containing:
- User interface
- Knowledge base format
- Inference engine

**Advantages:**
- Faster development
- Knowledge engineer focuses on knowledge, not infrastructure
- Can be customized for specific applications

**System Engineer:** Builds the shell, designs declarative format, implements inference engine (may be same person as knowledge engineer for small projects)

## 6. Comparison with Conventional Programs and Human Experts

| Characteristic | Human Experts | Expert Systems | Conventional Programs |
| --- | --- | --- | --- |
| Knowledge Type | Compiled, experiential | Rules and heuristics | Algorithms and data |
| Separation of Knowledge/Control | Integrated | Separate | Integrated |
| Explanation Ability | Can explain reasoning | Can trace rules fired | Cannot explain |
| Handling Uncertainty | Can handle incomplete/fuzzy data | Can handle incomplete/fuzzy data | Requires complete, exact data |
| Learning | Slow, expensive learning process | Add/modify rules easily | Changing code affects everything |
| Adaptability | Automatically adapts | Must be updated | Must be reprogrammed |

## 7. Advantages of Expert Systems

| Advantage                 | Description                                                   |

## 8. Disadvantages of Expert Systems

| Disadvantage                     | Description                                             |
| -------------------------------- | ------------------------------------------------------- |

## Two Primary Methods of Reasoning

The inference engine operates as the "brain" that processes knowledge. It takes:
- **A knowledge base:** A collection of facts, rules, and heuristics about a specific domain.
- **User input or current facts:** Information about the specific problem to be solved.

It then uses logical reasoning to combine these elements and derive conclusions.

| Method | Direction | Description |
| --- | --- | --- |
| Forward Chaining (Data-Driven) | Facts → Conclusion | Starts with known facts and applies rules to infer more facts until a goal is reached. Useful for monitoring, planning, and control tasks. |
| Backward Chaining (Goal-Driven) | Goal → Facts | Starts with a hypothesis (goal) and works backward to find evidence that supports it. Useful for diagnosis, consulting, and explanation tasks. |

## Example: Medical Diagnosis System
- **Knowledge base rule:** *IF* (fever AND cough AND difficulty breathing) *THEN* (suggest pneumonia).
- **Inference engine:**

In **forward chaining**: The engine receives inputs (fever = yes, cough = yes, difficulty breathing = yes) and activates the rule, concluding "pneumonia."
- In **backward chaining**: The engine tries to prove "pneumonia." It checks the rule's conditions and asks the user, "Do you have a fever? Do you have a cough? Do you have difficulty breathing?" to confirm or deny the hypothesis.

## Key Characteristics
- **Control mechanism:** It determines the order in which rules are evaluated and applied (conflict resolution).
- **Explanation capability:** It can trace the chain of reasoning to explain *why* a particular conclusion was reached—a critical feature for trust and transparency in AI systems.
- **Separation of concerns:** It separates domain knowledge (facts and rules) from the control logic, allowing the same engine to be used for different problems by simply swapping the knowledge base.

In summary, while the **knowledge base** represents *what* the system knows, the **inference engine** represents *how* the system thinks.

## 1. Introduction

Expert systems are intelligent programs that use domain-specific knowledge to solve complex problems. The architecture of an expert system consists of:
- User interface
- Knowledge base
- Database (working storage)
- Inference engine
- Explanation facility
- Knowledge acquisition facility

**Key Formula:** Expert System = Knowledge Base + Inference Engine

**Examples:**
- **MYCIN:** Medical diagnosis system
- **DENDRAL:** Determines molecular structure from chemical formula and data

**Why a Special Development Life Cycle?**
Conventional software development life cycles (SDLC) assume that requirements are well-defined upfront. Expert systems, however, deal with complex, often ill-defined problems where requirements evolve as understanding grows. Therefore, expert systems require **evolutionary** and **prototype-oriented** approaches developed through continuous interaction between the knowledge engineer and domain expert.

## 2. Stages in the Expert System Development Life Cycle (ESDLC)

The ESDLC consists of six iterative stages that cycle through knowledge acquisition, prototype critiquing, and prototype development until the system achieves desired stability.

## 2.1 Problem Selection

The first step is to identify and define the appropriate problem domain.

**Key Activities:**

| Activity | Description |
| --- | --- |
| Feasibility assessment | Determine if an expert system is the right approach and if it's worthwhile |
| Resource management | Assess and manage people, time, money, software, and hardware |
| Problem identification | Properly define the problem the client wants to solve |
| Expert identification | Identify domain experts who have sufficient knowledge and time |
| User identification | Identify the end users of the system |
| Approach selection | Identify appropriate problem-solving approach and discuss with experts |
| Sub-set selection | Select a sub-set of the problem for initial prototype development |
| Cost-benefit analysis | Evaluate economic feasibility |

**Outcome:** Initial Requirement Review Report (IRR), which includes:
- Problem description
- User skill specifications
- Established requirements
- Tentative prototype schedule
- Time constraints

## 2.2 Conceptualization

In this stage, the intended system capability is described, and the expertise needed for solving the problem is determined. The knowledge engineer works closely with the domain expert to clarify scope, identify sub-problems, and acquire knowledge.

**Two Main Tasks:**
- **Knowledge Source Identification and Selection**

Identify knowledge sources (experts, books, documents)
- Prioritize them by importance and availability
- Select appropriate sources

- **Knowledge Acquisition, Analysis, and Extraction**

Interview domain experts
- Ask questions about decisions, outcomes, resources, and conditions

**Key Questions for Experts:**
- Exactly what decisions does the expert make?
- What are the outcomes of these decisions?
- What resources or inputs are required to reach a decision?
- What conditions are present when a particular outcome is decided?

## 2.3 Formalization

Formalization involves organizing the acquired knowledge into a structured form suitable for implementation. Knowledge is classified into a hierarchical tree-like structure.

**Key Tasks:**

| Task | Objective |
| --- | --- |
| Knowledge representation | Specify how knowledge will be represented (rules, frames, logic) |
| Detailed control structure | Specify control structures (procedural embedding, rule groups, meta-level control) |
| Internal fact structure | Specify consistent structure for facts |
| Preliminary user interface | Design initial interface and get user feedback |
| Initial test plan | Define test data, drivers, and analysis procedures |
| Design structure | Specify logical organization of knowledge base |
| Implementation strategy | Specify how the system will be implemented |
| Detailed user interface | Refine interface based on feedback |
| Design specifications | Document the design |
| Detailed test plan | Specify exact testing and verification procedures |

## 2.4 Prototype Construction

**Definition of Prototyping (IEEE):** A type of development emphasizing early prototype creation to permit early feedback and analysis in support of the development process.

**Purpose:**
- Demonstrate system feasibility
- Allow domain expert feedback
- Reduce risk by identifying issues early
- Help users understand how the system will work

**Key Activities:**
- Map formalized knowledge into the development tool framework
- Implement core components (knowledge base, inference engine, database)
- Construct initial working model (not all features included)
- Demonstrate to domain expert for feedback and feasibility assessment
- Correct errors and refine understanding

**Types of Prototypes:**

| Type | Description | Purpose |
| --- | --- | --- |
| Throw-Away Prototype | Built with minimal requirements; discarded after requirements are validated | Elicit and validate requirements; reduce risk |
| Evolutionary Prototype | Evolves into final system through iterative refinement | Develop robust prototype that becomes the final system |

**Throw-Away Prototype Characteristics:**
- Unstructured and undocumented
- Does not specify non-functional requirements
- Steps: Gather requirements → Design prototype → Evaluate → Repeat → Write final requirements → Throw away

**Evolutionary Prototype Characteristics:**
- Properly structured
- Continuously refined by adding more knowledge
- Eventually becomes the final system

**Best Uses for Prototyping:**
- Growing list of requirements
- Web and application development
- Systems relying heavily on user input
- User interface development

**Advantages of Prototyping:**
- Cost-effective (changes early are cheaper)
- Better understanding of the proposed system
- Prevents misunderstandings between user and expert
- Provides feasibility feedback
- Increases user interaction
- Helps find incomplete or inconsistent requirements
- Reduces failure risk

**Disadvantages of Prototyping:**
- Users may confuse prototype with final system
- Security, backup, and recovery may be overlooked
- Documentation may be incomplete
- Quality may suffer due to haste
- Too many changes can disrupt development rhythm
- Developers may become attached to the prototype

## 2.5 Implementation

Implementation is the expanded prototyping phase where the initial prototype is debugged and expanded to its full capability.

**Key Activities:**

| Task | Objective |
| --- | --- |
| Coding | Implement coding of all features |
| Tests | Test code using test data and analysis procedures |
| Source listings | Produce commented, documented source code |
| User manual | Create working user manual for feedback |
| Installation/operations guide | Document installation and operation |
| System description document | Document functionality, limitations, and problems |

**Additional Activities:**
- Ensure user acceptance
- Install, demonstrate, and deploy the system
- Arrange user orientation and training
- Ensure security
- Provide documentation
- Arrange integration and field testing

## 2.6 Evaluation

The final stage involves thorough testing of all system components.

**Verification and Validation:**
- **Verification:** Checking that the system is built correctly (consistency, completeness)
- **Validation:** Checking that the right system was built (performance, utility)

**Areas Checked During Validation:**

| Area | Description |
| --- | --- |
| Rule consistency and completeness | Ensuring rules are logically sound and cover the domain |
| Rule interaction consequences | Avoiding unexpected outcomes from rule interactions |
| Information appropriateness | Ensuring explanations are clear and justified |
| Output agreement | Comparing system output with domain expert solutions |

**Post-Evaluation Activities:**
- Deliver the system and connect to existing communication network
- Implement security measures
- Begin maintenance (bug fixes and knowledge base updates)

## 3. Cyclic Nature of ESDLC

Expert system development is inherently iterative. The cycle of **knowledge acquisition → prototype critiquing → prototype development** repeats continuously throughout the life cycle.

**Key Insight:** Knowledge acquisition is a continuous process required in all stages, but with different purposes:
- Conceptualization: Gathering knowledge for building the database
- Evaluation: Refining the knowledge base
## 4. Sources of Error in Expert System Development

| Error Type              | Description                                                                                                                                                              |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Knowledge Errors        | Incorrect knowledge obtained from domain expert; leads to wrong decisions. Requires verification and validation of expert knowledge.                                     |
| Syntax Errors           | Rules or facts do not follow the syntax required by the development tool; occurs when knowledge engineer is unfamiliar with the tool.                                    |
| Semantic Errors         | Knowledge is incorrectly encoded into rules; rules do not reflect what the expert stated. May also occur if expert misinterprets questions.                              |
| Inference Engine Errors | Malfunctioning of the inference component (rule interactions or conflict resolution); leads to incorrect solutions.                                                      |
| Inference Chain Errors  | Errors arising from erroneous knowledge, inappropriate conclusions, semantic errors, inference engine bugs, inappropriate rule priorities, or strange rule interactions. |
