[[01 AI Usage]]
## 1. Начни проектировать систему до написания кода

Большинство разработчиков делают наоборот — сначала код.

Попробуй всегда делать так:

- Что это за система?

- Какие у неё модули?

- Как они общаются?

- Какие данные между ними передаются?

Простейший шаблон:

System
 ├─ API
 ├─ Database
 ├─ Worker
 └─ Frontend
Даже маленький проект проектируй так.

Это очень быстро прокачивает архитектурное мышление.

## 2. Делай архитектурные ревизии своего проекта

Очень мощный способ обучения.

Раз в неделю задавай себе вопросы:

- Где дублируется код?

- Что можно вынести в модуль?

- Где слишком сильная связность?

- Что можно сделать независимым?

Это и есть развитие абстракции.

## 3. Учись проектировать модели данных (это ключевой навык)

Это то, что автор подкаста считает самым важным.

Практика:

Перед кодом пиши:

User
- id
- email
- role
- created_at
И только потом код.

Профессиональные разработчики делают именно так.

## 4. Делай маленькие архитектуры каждый день

Лучшее упражнение:

Возьми любой сервис и попробуй его спроектировать.

Например:

- Reddit

- Notion

- Uber

- Telegram

- GitHub

Спроси себя:

- какие сервисы внутри?

- какие базы данных?

- где кеш?

- где очередь?

Это реально тренирует мозг.

## 5. Учись выделять изолированные части системы

Это то, что сильно связано с AI-разработкой.

Хороший вопрос:

Можно ли удалить этот модуль и переписать его за 1 день?

Если нет — архитектура слабая.

## 6. Используй AI правильно (очень важный момент)

Большая ошибка сейчас:

люди дают AI писать всё.

Лучший подход:

ты проектируешь → AI реализует.

Пример правильного workflow:

1. Спроектировать модель
2. Спроектировать интерфейсы
3. Разделить систему
4. Дать AI писать реализацию
Это и есть то, о чём говорит автор.

## Самое эффективное упражнение (очень сильное)

Сделай один проект так:

- Архитектура на бумаге

- Data models

- API контракты

- Только потом код

Это буквально меняет уровень разработчика.

## Если честно: самый быстрый путь

Вот реальная формула роста:

1 сложный проект
+ регулярный рефакторинг
+ анализ архитектуры
= сильный developer
Не курсы.

## Маленький тест (покажет уровень архитектора)

Попробуй ответить:

Как бы ты построил систему вроде Obsidian Sync?

Нужно:

- синхронизация

- версии файлов

- офлайн режим

- конфликты

Если хочешь, можем разобрать это — это очень хорошая тренировка для system design.
